# C. 안티패턴 카탈로그 (확장판)

> Part 1.2의 5가지 안티패턴 + 추가 패턴 5가지 = 총 10개

각 패턴: **증상 / 원인 / 해법 / 관련 챕터**

## 핵심 5가지 (Part 1.2)

### 1. 전부 맡기기

- **증상**: AI가 만든 코드가 컴파일도 안 됨
- **원인**: AI에 큰 작업을 통째로 던짐, 컨텍스트·역할 분담 부재
- **해법**: AI가 못하는 부분을 도구·템플릿이 맡음 (Part 2.2 우아한형제들 사례)
- **관련**: [2.2 Plan-based Execution](../part-2-what/plan-based-execution)

### 2. 컨텍스트 없이 프롬프트만

- **증상**: 같은 설명을 매번 반복, AI가 매번 같은 실수
- **원인**: 프로젝트 컨텍스트 파일 부재
- **해법**: CLAUDE.md / AGENTS.md
- **관련**: [2.1 Context Engineering](../part-2-what/context-engineering)

### 3. 플랜 없이 즉시 실행

- **증상**: 10분 뒤 "방향 다른데" → 롤백
- **원인**: 요구사항이 불명확한 채 실행
- **해법**: Plan Mode로 합의 후 실행
- **관련**: [2.2 Plan-based Execution](../part-2-what/plan-based-execution)

### 4. 검증 없이 완료 선언

- **증상**: AI가 "완료"라고 했는데 며칠 뒤 프로덕션 장애
- **원인**: 완료의 정의가 사람 머릿속에만 있음
- **해법**: TDD, 훅, 리뷰 에이전트
- **관련**: [2.4 Quality Verification](../part-2-what/quality-verification)

### 5. 하나의 에이전트에 다 몰기

- **증상**: 같은 파일을 10번째 다시 읽음, 자기가 만든 버그를 자기가 못 찾음
- **원인**: 컨텍스트 오염, 역할 혼재
- **해법**: 서브에이전트로 격리, Plan/Impl/Review 분리
- **관련**: [2.3 Token Optimization](../part-2-what/token-optimization), [2.5 Multi-Agent](../part-2-what/multi-agent)

## 추가 5가지 (자주 보는 디테일)

### 6. "AI 결과물 그대로 PR"

- **증상**: 리뷰어가 코드를 처음 보는 사람이 됨, AI 흔적이 곳곳에 남음
- **원인**: 사람의 검토·정제 단계 생략
- **해법**: AI 결과물은 **초안**이지 **PR**이 아니다 — 사람이 한 번 정제 후 제출
- **관련**: [2.4 품질 검증](../part-2-what/quality-verification)

### 7. "프롬프트 골든 카피" 의존

- **증상**: 누군가가 만든 "잘 먹히는 프롬프트"를 모두가 복붙해서 씀
- **원인**: 원리 이해 없이 결과만 베끼기
- **해법**: 프롬프트가 **왜** 잘 먹히는지 이해하기, 본인 맥락에 맞게 변형
- **관련**: 모든 챕터

### 8. "AI 면죄부"

- **증상**: 버그가 나면 "AI가 그렇게 썼어요"
- **원인**: 코드 책임을 AI에 전가
- **해법**: **사람이 작성자다**. AI는 도구. 책임은 PR을 올린 사람.
- **관련**: [2.4 품질 검증](../part-2-what/quality-verification)

### 9. "한 번 만든 CLAUDE.md 방치"

- **증상**: CLAUDE.md가 6개월 전 내용 그대로
- **원인**: 죽은 문서. 운영 책임자 없음.
- **해법**: PR마다 한 번 질문 — "이게 CLAUDE.md에 반영될 만한 규칙인가?"
- **관련**: [3.1 CLAUDE.md 시작하기](../part-3-apply/starting-claude-md)

### 10. "도구만 바꾸면 다 해결"

- **증상**: Cursor → Claude Code → Codex → ... 도구 옮기는 데만 시간 씀
- **원인**: 문제가 도구가 아니라 **하네스**에 있다는 걸 모름
- **해법**: 도구를 바꾸기 전에 **5가지 축**을 점검
- **관련**: [2.0 하네스 엔지니어링](../part-2-what/harness-engineering)
