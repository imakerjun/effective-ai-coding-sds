# 3.1 CLAUDE.md를 시작하는 법

> 완벽한 CLAUDE.md는 존재하지 않습니다. **자라는** CLAUDE.md가 있을 뿐입니다.

## 흔한 첫 실수

많은 팀이 CLAUDE.md를 도입할 때 같은 실수를 합니다:

- ❌ "일단 완벽하게 써보자" → 한 달째 빈 파일
- ❌ 모든 컨벤션을 한 번에 넣으려 함 → 너무 길어서 아무도 안 읽음
- ❌ README와 중복 → 관리 부담 2배
- ❌ 한 번 만들고 방치 → 6개월 뒤 현실과 괴리

**"완벽한 첫 버전"을 만들려 하지 마세요. 20분 안에 1버전을 만들고, 계속 고쳐나가는 게 정답입니다.**

## 20분 템플릿

아래를 그대로 복사해서 시작하세요. 빈 칸을 채우는 데 **20분 이상 쓰지 마세요**.

```markdown
# [프로젝트명]

## 이 레포가 뭐 하는 건지 (1~2줄)
[여기에]

## 디렉터리 구조 (주요 폴더만)
- src/ — 메인 코드
- tests/ — 테스트
- docs/ — 문서

## 코딩 컨벤션 (지금 당장 지켜야 할 것만 5개)
1.
2.
3.
4.
5.

## AI가 자주 헤매는 것 (매우 중요)
- 지난주에 AI가 틀렸던 것 1:
- 지난주에 AI가 틀렸던 것 2:
- 지난주에 AI가 틀렸던 것 3:

## 건드리면 안 되는 것
- [예: legacy/ 폴더]
- [예: 자동 생성 파일]

## 주요 명령어
| 명령어 | 설명 |
|---|---|
| npm run dev | 개발 서버 |
| npm test | 테스트 |

## 커밋 컨벤션
- [예: feat/fix/chore 접두사]
```

## 각 섹션을 제대로 쓰는 법

### "AI가 자주 헤매는 것" — 가장 가치 있는 섹션

이 섹션이 **CLAUDE.md에서 가장 중요**합니다. 이유는:

- 컨벤션은 다른 문서에도 있을 수 있다
- 명령어는 `package.json`에 있다
- 하지만 **"AI가 매번 헤매는 것"** 은 여기밖에 안 담긴다

**쓰는 방법**:
- 지난 1~2주 AI와의 대화에서 **반복해서 틀렸던 것**을 적는다
- "왜 틀리는지" 짧게 설명 (AI가 배경을 이해하도록)
- 가급적 **예시 포함** ("이건 X처럼 하지 말고 Y처럼")

**예시**:
```markdown
## AI가 자주 헤매는 것

### 1. Kotlin에서 data class 쓰지 마세요
이 프로젝트는 불변 객체 패턴을 위해 sealed interface를 씁니다.
예: `sealed interface User { ... }` (data class ❌)

### 2. 테스트 파일 위치
`src/test/kotlin/` 아래, 원본과 같은 패키지 구조로.
루트에 TestXxx.kt 만들지 마세요.

### 3. 에러 메시지는 한글
사용자 노출 에러 메시지는 한글, 로그 메시지는 영어.
```

### "건드리면 안 되는 것"

**위험 영역을 명시**하면 AI가 의도치 않게 건드리는 것을 크게 줄일 수 있습니다.

```markdown
## 건드리면 안 되는 것

- `legacy/` — 레거시 코드, 리팩터링 금지
- `db/migrations/*` — 기존 마이그레이션 파일은 수정 금지 (새 파일만 추가)
- `.env*` — 환경 변수 파일 읽기/쓰기 금지
- `third_party/` — 외부 라이브러리 복사본
```

### 컨벤션은 "5개만"

한 번에 50개 쓰면 안 읽힙니다. **5개만** 쓰세요. 진짜 중요한 것만.

- 우선순위 1: 최근 PR에서 리뷰 지적이 나온 것
- 우선순위 2: 팀에서 합의된 금지사항
- 우선순위 3: 시니어가 "이건 꼭 지켜야 해"라고 생각하는 것

## 시작 후 1주일 동안 할 것

### Day 1: 20분 초안

위 템플릿 채우기. 완벽하지 않아도 커밋.

### Day 2~7: 관찰 모드

- AI 에이전트에게 일을 시키면서 **"또 헤매네"** 싶은 순간 포착
- 그 순간마다 CLAUDE.md에 규칙 추가
- 매일 1~2개씩만 추가해도 1주일이면 풍부해짐

### Day 7: 리뷰

- 팀원과 함께 CLAUDE.md를 **소리 내어 읽기**
- 중복·모순·불명확한 부분 수정
- 팀 공유 합의 → main 브랜치 반영

## 이후 운영 원칙

### ✅ 하면 좋은 것

- **PR마다 한 번 질문**: "이 변경이 CLAUDE.md에 반영될 만한 규칙을 만드는가?"
- AI가 또 같은 실수를 하면 → 그 자리에서 규칙 추가
- 월 1회 정리 — 오래되고 안 쓰이는 규칙 삭제

### ❌ 하지 말 것

- **복사해서 여러 저장소에 방치**하지 말 것 — 각 프로젝트는 다름
- **100줄 넘게 키우지 말 것** — 길어지면 부록 문서로 분리
- **한 번 만들고 잊지 말 것** — 안 자라는 CLAUDE.md는 죽은 문서

## 🤖 AI Pro에서는?

같은 챕터를 AI Pro에서 진행할 때:

> **AI Pro에서는 1차 컨텍스트 파일이 `AGENTS.md`** 입니다 (PDF 가이드 명시). Rules는 행동 지침을 보완하는 역할입니다.

| 단계 | Claude Code | AI Pro |
|---|---|---|
| 1. 자동 초안 받기 | `/init` → `CLAUDE.md` 생성 | `/init` → `AGENTS.md` 생성 |
| 2. 위 20분 템플릿으로 다듬기 | `CLAUDE.md` 편집 | `AGENTS.md` 편집 |
| 3. 행동 지침 보강 | (없음) | `.aipro/rules/main.md` 추가 (선택) |
| 4. 인식 테스트 | `claude` 실행 후 "이 프로젝트 컨벤션 알고 있어?" | AI Pro Chat에서 동일하게 질문 |

**AI Pro 전용 팁**:

- **AGENTS.md를 1차로 채우세요**: `AGENTS.md`가 프로젝트 컨텍스트의 메인입니다. Rules는 "AI Pro가 항상 따라야 할 행동 방식"이 따로 있을 때만 추가하세요.
- **Rules는 합산 10,000자 제한**: AGENTS.md와 별개로, Rules는 합쳐 10,000자까지만 LLM에 전달됩니다. 컨벤션이 길어지면 AGENTS.md로 옮기세요.
- **CLI + IDE 동시 사용**: `AGENTS.md`는 CLI/IDE 모두에 동일하게 적용됩니다.
